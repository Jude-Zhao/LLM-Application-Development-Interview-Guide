# 后端开发八股文（通用面试向）

> 整理时间：2026-03-19  
> 适用岗位：后端开发、服务端开发、平台研发、业务后端开发  
> 说明：这份内容按“面试可直接回答”的风格整理，重点放在常见高频问题和工程上的真实取舍，不追求底层源码推导。

---

## 1. 这个岗位通常会问什么

后端面试高频模块通常集中在这几类：

- 计算机网络：`HTTP`、`HTTPS`、`TCP`、长连接、状态码
- 数据库：索引、事务、隔离级别、`MVCC`、慢查询优化
- 缓存：`Redis`、缓存穿透、击穿、雪崩、一致性
- 消息队列：削峰、解耦、异步、重复消费、顺序消费
- 并发编程：线程、进程、协程、锁、线程池
- 分布式：负载均衡、分库分表、幂等、限流、熔断
- 工程实践：日志、监控、告警、灰度、回滚、压测

一句话：

后端面试不只是问“概念是什么”，更关注你知不知道这些东西为什么要用、解决什么问题、出了问题怎么排查。

## 2. 面试高频背诵版

### 2.1 `GET` 和 `POST` 的区别

- `GET` 更常用于获取资源，参数通常放在 URL 中。
- `POST` 更常用于提交数据，参数通常放在请求体中。
- `GET` 更容易被缓存、收藏、记录在浏览器历史中。
- 从 HTTP 语义上说，`GET` 应该是幂等的，`POST` 不一定。

注意：

- `GET` 不是绝对不能带请求体，只是实际使用中很少这样设计。
- `POST` 也不是天然不安全，安全性主要看 `HTTPS`、鉴权和服务端校验。

一句话回答：

`GET` 偏查询、应当幂等；`POST` 偏提交、通常用于产生副作用的操作。

### 2.2 `HTTP` 和 `HTTPS` 的区别

- `HTTP` 明文传输。
- `HTTPS` = `HTTP + TLS`，会对传输内容做加密。
- `HTTPS` 可以同时提供：
  - 机密性
  - 完整性
  - 身份认证

一句话回答：

`HTTPS` 本质是在 `HTTP` 之下加了一层 `TLS`，解决明文传输和中间人攻击问题。

### 2.3 `HTTPS` 握手大致做了什么

你面试不用背完整报文，讲清楚这几步就够：

- 客户端和服务端协商加密套件
- 服务端发送证书
- 客户端验证证书是否合法
- 双方协商出会话密钥
- 后续使用对称加密传输数据

一句话回答：

`HTTPS` 握手的目标是验证服务端身份，并安全地协商出后续通信使用的密钥。

### 2.4 `TCP` 为什么可靠

`TCP` 的可靠性主要来自：

- 三次握手建立连接
- 序号机制保证顺序
- 确认应答 `ACK`
- 超时重传
- 流量控制
- 拥塞控制

一句话回答：

`TCP` 通过确认、重传、排序和控制机制，保证数据尽量可靠、有序地送达。

### 2.5 为什么是三次握手，不是两次

核心原因是双方都要确认：

- 自己的发送能力正常
- 自己的接收能力正常
- 对方的发送能力正常
- 对方的接收能力正常

如果只有两次，服务端无法确认客户端是否具备接收能力。

一句话回答：

三次握手的本质是让双方都确认“收发能力正常”，两次不够完整。

### 2.6 为什么挥手通常是四次

因为 `TCP` 是全双工连接：

- 一方说“我没有数据发了”
- 另一方可能还有数据没发完

所以关闭发送和确认关闭通常要分开。

一句话回答：

四次挥手是因为连接两端的关闭动作通常不能合并，双方都要独立完成关闭。

### 2.7 什么是长连接

长连接指：

- 一次建连后，不是请求完马上断开
- 后续多个请求复用同一条连接

好处：

- 减少频繁握手带来的开销
- 降低延迟
- 提升吞吐

一句话回答：

长连接的核心价值是复用连接，减少反复建连的成本。

### 2.8 `Cookie`、`Session`、`JWT` 的区别

#### Cookie

- 存在客户端
- 浏览器会自动携带

#### Session

- 会话数据通常存在服务端
- 客户端只保存 `session_id`

#### JWT

- 常见做法是把用户信息和签名放进一个 token
- 服务端可以无状态校验

一句话回答：

`Cookie` 是载体，`Session` 是服务端会话方案，`JWT` 是一种常见的无状态认证 token 方案。

### 2.9 什么是幂等

幂等指的是：

- 同一个请求执行一次和执行多次，最终结果一致

常见场景：

- 重试
- 超时补偿
- 消息重复消费
- 支付回调

常见实现：

- 唯一业务单号
- 去重表
- 幂等 token
- 状态机约束

一句话回答：

幂等的本质是防止重复请求把系统状态改乱。

### 2.10 什么是数据库索引

索引可以理解为：

- 帮助数据库快速定位数据的数据结构

常见好处：

- 提升查询速度
- 减少全表扫描

常见代价：

- 占空间
- 会影响写入性能

一句话回答：

索引本质是用空间换时间，让查询更快，但会增加维护成本。

### 2.11 为什么 MySQL 常说用 `B+` 树索引

面试里通常回答这几个点：

- 树高低，磁盘 IO 次数少
- 非叶子节点只存索引，更适合做范围查询
- 叶子节点有序，天然适合顺序扫描

一句话回答：

`B+` 树兼顾了磁盘访问效率、范围查询能力和排序扫描能力，所以很适合数据库索引。

### 2.12 聚簇索引和二级索引的区别

以 InnoDB 为例：

- 聚簇索引的叶子节点存的是整行数据
- 二级索引的叶子节点存的是索引列值和主键值

因此通过二级索引查整行数据时，通常还要回表。

一句话回答：

聚簇索引直接存数据，二级索引通常先定位主键，再决定是否回表取整行。

### 2.13 什么情况下索引会失效

常见情况：

- 对索引列做函数操作
- 隐式类型转换
- 最左前缀没用上
- 范围查询后，后续列利用不足
- `like '%xxx'` 不能有效走前缀索引

一句话回答：

索引失效本质上是数据库没法继续利用索引的有序性和可定位性。

### 2.14 什么是事务

事务可以理解为：

- 一组操作要么都成功，要么都失败

它的核心特性是 `ACID`：

- 原子性
- 一致性
- 隔离性
- 持久性

一句话回答：

事务的目标是把多步操作变成一个可靠的整体。

### 2.15 数据库隔离级别有哪些

常见四种：

- 读未提交
- 读已提交
- 可重复读
- 串行化

隔离级别越高：

- 一般并发问题越少
- 但性能和并发能力通常会下降

一句话回答：

隔离级别是在“并发性能”和“数据一致性”之间做权衡。

### 2.16 脏读、不可重复读、幻读分别是什么

#### 脏读

- 读到了别人还没提交的数据

#### 不可重复读

- 同一事务里，两次读取同一行，结果不一样

#### 幻读

- 同一事务里，两次按条件查询，记录条数不一样

一句话回答：

脏读是读到未提交数据，不可重复读是同一行前后不一致，幻读是结果集条数发生变化。

### 2.17 什么是 `MVCC`

`MVCC` 是多版本并发控制。

你可以这样理解：

- 一行数据不只一个版本
- 读操作根据版本规则看到合适的数据快照
- 这样读写可以减少互相阻塞

一句话回答：

`MVCC` 的核心价值是用“多版本快照”提升并发读写能力。

### 2.18 乐观锁和悲观锁的区别

#### 乐观锁

- 假设冲突少
- 提交更新时再检查版本号或条件

#### 悲观锁

- 假设冲突多
- 先加锁再操作

一句话回答：

乐观锁适合读多写少、冲突少；悲观锁适合冲突明显、必须强保护的场景。

### 2.19 Redis 为什么快

常见回答：

- 基于内存
- 数据结构高效
- 单线程模型避免了大量锁竞争
- 网络模型和实现比较轻量

一句话回答：

Redis 快的根本原因是“以内存为主 + 实现轻量 + 少锁竞争”。

### 2.20 Redis 常见数据结构有哪些

- `String`
- `Hash`
- `List`
- `Set`
- `ZSet`
- `Bitmap`
- `HyperLogLog`
- `Stream`

面试通常要能讲清楚：

- 什么时候用 `String`
- 什么时候用 `Hash`
- 什么时候用 `ZSet`

### 2.21 什么是缓存穿透、击穿、雪崩

#### 缓存穿透

- 请求的数据本来就不存在
- 每次都打到数据库

常见解决：

- 缓存空值
- 布隆过滤器

#### 缓存击穿

- 某个热点 key 突然失效
- 大量请求同时打到数据库

常见解决：

- 热点 key 永不过期
- 加互斥锁

#### 缓存雪崩

- 大量 key 在同一时间失效
- 数据库压力瞬间被打爆

常见解决：

- 过期时间加随机值
- 多级缓存
- 限流降级

一句话回答：

穿透是“查不存在”，击穿是“热点失效”，雪崩是“集体失效”。

### 2.22 如何保证缓存和数据库一致性

完全强一致通常代价很高，工程里常见是“尽量最终一致”。

常见方案：

- 先更新数据库，再删除缓存
- 配合重试和消息补偿
- 关键场景加延迟双删

一句话回答：

缓存和数据库很难低成本做到绝对强一致，工程上更常做的是控制不一致窗口并补偿。

### 2.23 为什么要用消息队列

核心价值通常有三个：

- 解耦
- 异步
- 削峰

一句话回答：

消息队列的本质是把“直接同步调用”改成“异步解耦的事件流转”。

### 2.24 消息队列会带来哪些问题

常见问题：

- 消息丢失
- 消息重复
- 消息积压
- 顺序问题
- 消费失败怎么重试

一句话回答：

用了消息队列之后，不是问题消失了，而是把同步问题换成了异步一致性问题。

### 2.25 如何保证消息不丢

需要分生产端、MQ 本身、消费端来看：

- 生产者发送成功确认
- MQ 做持久化
- 消费成功后再确认消费
- 失败重试和死信队列

一句话回答：

消息“不丢”不是单点保证，而是生产、存储、消费三段都要做好确认和持久化。

### 2.26 如何避免消息重复消费

不能只指望 MQ 不重复，消费端通常也要做幂等。

常见方式：

- 唯一消息 ID 去重
- 去重表
- 业务状态判断
- 唯一索引约束

一句话回答：

重复消费通常要靠“消费端幂等”兜底，而不是只相信 MQ 恰好只投一次。

### 2.27 什么是限流

限流就是：

- 控制单位时间内允许进入系统的请求量

常见算法：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

一句话回答：

限流的目标是保护系统，不让流量在短时间内把服务打垮。

### 2.28 什么是熔断和降级

#### 熔断

- 下游异常率过高时，先快速失败，避免继续拖垮系统

#### 降级

- 在系统压力大或依赖异常时，主动返回兜底结果

一句话回答：

熔断是“先断开坏依赖”，降级是“用差一些但可接受的结果保住可用性”。

### 2.29 什么是负载均衡

负载均衡就是：

- 把请求分发到多台机器上
- 提升吞吐、可用性和扩展能力

常见策略：

- 轮询
- 加权轮询
- 最少连接
- 一致性哈希

### 2.30 什么是分库分表

当单库单表扛不住时，会考虑拆分。

目的通常是：

- 降低单表数据量
- 分散读写压力
- 提高扩展性

代价通常是：

- 跨库事务复杂
- 跨分片查询麻烦
- 运维复杂度上升

一句话回答：

分库分表是典型的“用复杂度换容量和性能”的方案。

### 2.31 什么是分布式事务

分布式事务指：

- 一次业务操作涉及多个服务或多个存储节点
- 需要保证整体一致性

常见思路：

- 两阶段提交
- `TCC`
- 本地消息表
- 最终一致性补偿

一句话回答：

分布式事务的核心难点是多个独立节点之间很难低成本做到强一致。

### 2.32 什么是 `CAP`

`CAP` 说的是分布式系统中：

- 一致性 `C`
- 可用性 `A`
- 分区容错性 `P`

在发生网络分区时，通常无法同时完美满足 `C` 和 `A`。

一句话回答：

`CAP` 不是让你三选二做背诵题，而是提醒你分布式系统必须面对网络分区带来的取舍。

### 2.33 线程、进程、协程的区别

#### 进程

- 资源隔离更强
- 切换成本更高

#### 线程

- 同一进程内共享内存
- 比进程更轻量

#### 协程

- 用户态调度
- 切换成本更低
- 特别适合高并发 IO

一句话回答：

进程重、线程中等、协程最轻，后端高并发 IO 场景通常会重点用协程或异步模型。

### 2.34 什么是线程池，为什么要用线程池

线程池的核心价值：

- 复用线程
- 限制并发数
- 降低频繁创建销毁线程的成本

一句话回答：

线程池本质上是在“资源可控”的前提下复用线程，提高系统稳定性。

### 2.35 什么是连接池

连接池和线程池很像，本质都是资源池化。

常见对象：

- 数据库连接池
- Redis 连接池
- HTTP 连接池

核心价值：

- 减少频繁建连开销
- 控制资源上限

### 2.36 慢 SQL 怎么排查

常见思路：

1. 先看执行计划
2. 看是否命中索引
3. 看扫描行数是否过大
4. 看是否有回表、排序、临时表
5. 看 SQL 写法和业务条件是否合理

一句话回答：

慢 SQL 排查不能只盯 SQL 语句本身，要结合执行计划、索引、数据量和业务访问模式一起看。

### 2.37 如何设计一个高并发接口

常见回答框架：

- 前面做限流
- 服务做无状态扩容
- 热点数据走缓存
- 非核心链路异步化
- 做监控告警和降级兜底

一句话回答：

高并发设计的本质是把“所有请求都打到核心资源”变成“分层消化和限量放行”。

## 3. 项目表达时常见追问

### 3.1 线上接口 RT 很高，你怎么排查

可以按链路拆：

1. 看监控，确认是单接口还是全链路问题
2. 看应用日志和调用链，定位卡在应用、DB、Redis 还是下游服务
3. 看机器资源：CPU、内存、线程池、连接池
4. 看数据库慢查询和缓存命中率
5. 看是否有流量突增、发布变更或依赖故障

面试表达重点：

- 排查要有顺序
- 先定位瓶颈层
- 再决定扩容、优化、降级还是回滚

### 3.2 如果缓存挂了怎么办

常见回答：

- 先限流，避免数据库被瞬间打穿
- 打开降级开关，非核心数据允许返回兜底
- 热点数据优先恢复
- 必要时临时扩容数据库和服务实例

面试重点：

- 不要只说“重启 Redis”
- 要说流量保护和核心链路保护

### 3.3 如何做接口幂等

常见说法：

- 请求带唯一业务单号
- 服务端落幂等表或唯一索引
- 重复请求直接返回第一次结果或当前状态

这类题特别适合结合：

- 支付
- 下单
- 回调通知
- MQ 消费

### 3.4 数据库压力大怎么优化

可以从四层讲：

1. SQL 优化
2. 索引优化
3. 缓存分流
4. 读写分离、分库分表

面试里不要一上来就说分库分表。

因为很多问题在那之前可以先通过：

- 索引
- SQL 重写
- 缓存
- 批处理

解决掉。

## 4. 面试里的回答套路

### 4.1 回答概念题的通用模板

建议按这四步答：

1. 先说定义
2. 再说解决什么问题
3. 再说核心优缺点
4. 最后补一个实际使用场景

例如回答 Redis：

- 它是什么
- 为什么快
- 适合做什么
- 有什么风险和限制

### 4.2 回答线上问题排查题的通用模板

建议按这四步答：

1. 先止血
2. 再定位
3. 再修复
4. 最后复盘

这样会显得你不是只会背概念，而是有工程处理意识。

### 4.3 回答系统设计题的通用模板

建议至少覆盖：

- 业务目标
- 核心链路
- 存储选型
- 缓存方案
- 一致性方案
- 高可用和容灾
- 监控与压测

## 5. 最后怎么准备更有效

如果你是校招或初中级后端，优先顺序建议是：

1. `HTTP / TCP / HTTPS`
