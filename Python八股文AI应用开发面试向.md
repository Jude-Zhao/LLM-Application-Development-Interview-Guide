## 1. 面试高频背诵版

### 1.1 `list` 和 `tuple` 的区别

- `list` 是可变类型，`tuple` 是不可变类型。
- `list` 更适合存放一组会变化的数据；`tuple` 更适合表示一组结构固定的数据。
- `tuple` 因为不可变，通常可以作为字典 key 的一部分，但前提是内部元素也必须可哈希。
- 面试里常见追问：`tuple` 自己不可变，但它里面可以引用可变对象。

一句话回答：

`list` 适合“要改”的数据，`tuple` 适合“结构固定”的数据；核心区别是可变性。

### 1.2 可变对象和不可变对象有哪些

- 不可变：`int`、`float`、`str`、`tuple`、`bool`
- 可变：`list`、`dict`、`set`

注意：

- “不可变”说的是对象本身不能原地修改。
- 如果 `tuple` 里放了 `list`，那 `tuple` 里的元素引用不能变，但那个 `list` 里的内容还是能变。

### 1.3 `==` 和 `is` 的区别

- `==` 比较的是值是否相等。
- `is` 比较的是两个变量是不是同一个对象，也就是对象身份是否相同。

一句话回答：

`==` 看值，`is` 看是不是同一个对象。

### 1.4 Python 赋值是拷贝吗

不是。

- Python 的赋值本质上是“名字绑定到对象”。
- `b = a` 不会复制对象，只是让 `b` 和 `a` 指向同一个对象。

示例：

```python
a = [1, 2]
b = a
b.append(3)
print(a)  # [1, 2, 3]
```

一句话回答：

赋值不是拷贝，只是多了一个变量名指向同一个对象。

### 1.5 浅拷贝和深拷贝的区别

- 浅拷贝：只拷贝最外层对象，内部嵌套对象仍然共享引用。
- 深拷贝：递归拷贝，内层对象也会复制一份。

常见写法：

```python
import copy

b = copy.copy(a)      # 浅拷贝
c = copy.deepcopy(a)  # 深拷贝
```

一句话回答：

浅拷贝只复制外壳，深拷贝会把嵌套对象也一起复制。

### 1.6 为什么不建议把可变对象写成默认参数

因为函数默认参数只在函数定义时求值一次，不是每次调用都新建。

错误示例：

```python
def add_item(x, items=[]):
    items.append(x)
    return items
```

推荐写法：

```python
def add_item(x, items=None):
    if items is None:
        items = []
    items.append(x)
    return items
```

一句话回答：

可变默认参数会在多次调用之间共享，容易产生脏数据。

### 1.7 Python 函数传参到底是什么机制

更准确的说法不是“值传递”也不是“引用传递”，而是：

- 传的是对象引用
- 形参和实参最终绑定到同一个对象

所以：

- 如果对象是可变的，在函数内部原地修改，外部能看到变化。
- 如果在函数内部重新赋值，只是让形参重新绑定，不会影响外部变量名。

### 1.8 `*args` 和 `**kwargs` 是什么

- `*args` 接收多余的位置参数，类型是 `tuple`
- `**kwargs` 接收多余的关键字参数，类型是 `dict`

示例：

```python
def func(a, *args, **kwargs):
    print(a, args, kwargs)
```

面试回答重点：

- 用于写通用函数、装饰器、框架代码时很常见。

### 1.9 `global` 和 `nonlocal` 的区别

- `global`：在函数内部声明使用模块级全局变量。
- `nonlocal`：在嵌套函数内部，声明使用外层函数里的变量。

一句话回答：

`global` 改模块级变量，`nonlocal` 改最近一层闭包变量。

### 1.10 闭包是什么，为什么循环里 lambda 容易出问题

闭包可以理解为：

- 内部函数引用了外部函数作用域中的变量
- 外部函数执行完后，这些变量仍然能被内部函数使用

循环里 lambda 常见坑：

```python
funcs = []
for x in range(3):
    funcs.append(lambda: x)

print(funcs[0]())  # 2
```

原因：

- 闭包里保存的是变量，不是当时的值
- 等真正执行时，`x` 已经变成循环结束后的值

常见修复：

```python
funcs.append(lambda x=x: x)
```

### 1.11 装饰器是什么

装饰器本质上是：

- 一个接收函数并返回新函数的函数
- 用来在不改原函数源码的前提下增强功能

典型用途：

- 日志
- 权限校验
- 耗时统计
- 重试
- 缓存

一句话回答：

装饰器就是对函数做包装，常用于横切逻辑。

### 1.12 迭代器和生成器的区别

迭代器：

- 实现了迭代器协议，核心是 `__iter__()` 和 `__next__()`

生成器：

- 是一种更方便的迭代器实现方式
- 只要函数里有 `yield`，这个函数调用后就会返回生成器对象

一句话回答：

生成器是一种特殊迭代器，写法更简单，还支持惰性计算。

### 1.13 生成器有什么好处

- 节省内存，不需要一次性把所有数据都加载进来
- 适合大文件处理、流式返回、逐条数据处理
- 很适合 AI 应用里的流式输出场景

### 1.14 `with` 语句的作用是什么

- `with` 用于上下文管理
- 核心是对象实现了 `__enter__()` 和 `__exit__()`
- 作用是把资源申请和释放的逻辑包装起来

典型场景：

- 文件操作
- 数据库连接
- 锁
- 网络连接

一句话回答：

`with` 能保证资源正确释放，本质依赖上下文管理器协议。

### 1.15 `sort()` 和 `sorted()` 的区别

- `list.sort()`：原地排序，返回 `None`
- `sorted()`：返回新列表，不修改原对象

一句话回答：

要保留原数据用 `sorted()`；只想原地排序用 `sort()`。

### 1.16 `dict` 的底层是什么，为什么查找快

- CPython 中 `dict` 底层是可扩容哈希表
- 通过 key 的哈希值定位存储位置，所以平均查找效率高

常见补充：

- Python 3.7+ 保证字典保持插入顺序
- 字典 key 一般要求可哈希，通常也应当不可变

### 1.17 为什么字典的 key 一般要求不可变

因为 key 要参与哈希。

如果 key 可变：

- 它的哈希值或相等性行为可能变化
- 会破坏字典查找和存储的一致性

所以常见 key：

- `str`
- `int`
- `tuple`（前提是内部元素也可哈希）

### 1.18 `set` 的特点是什么

- 无序、元素唯一
- 底层同样基于哈希思想
- 常用于去重、集合运算、成员判断

面试一句话：

`set` 最常用的价值就是去重和高效判断元素是否存在。

### 1.19 `try/except/finally` 的执行顺序

- `try` 先执行
- 如果报错，匹配到对应的 `except` 执行
- 不管是否报错，`finally` 一般都会执行

面试重点：

- 没异常时，`try -> finally`
- 有异常且被捕获时，`try -> except -> finally`

### 1.20 Python 的垃圾回收机制是什么

常见回答：

- CPython 主要使用引用计数
- 同时配合垃圾回收器处理循环引用

补充：

- 不能完全依赖对象被回收时自动释放外部资源
- 文件、网络连接、数据库连接最好显式关闭，或使用 `with`

### 1.21 GIL 是什么

GIL 是 CPython 里的全局解释器锁。

含义：

- 同一时刻只有一个线程在执行 Python 字节码

影响：

- CPU 密集型任务，多线程通常不能真正利用多核
- I/O 密集型任务，多线程仍然有价值

一句话回答：

GIL 限制了 CPython 多线程的 CPU 并行能力，但不妨碍它做 I/O 并发。

### 1.22 多线程、多进程、协程怎么选

#### 多线程

- 适合 I/O 密集型
- 共享内存方便
- 受 GIL 影响，不适合 CPU 密集型并行

#### 多进程

- 适合 CPU 密集型
- 可以绕过 GIL
- 进程间通信成本更高

#### 协程 / `asyncio`

- 单线程并发
- 适合大量网络请求、接口编排、长连接、Web 服务
- 不是并行，适合 I/O 等待多的场景

面试一句话：

I/O 密集型优先线程或协程，CPU 密集型优先多进程。

### 1.23 `asyncio` 是什么

- `asyncio` 是 Python 官方异步 I/O 库
- 基于 `async/await` 写并发代码
- 适合高并发网络请求、异步 Web、异步数据库访问等场景

一句话回答：

`asyncio` 适合把大量等待 I/O 的任务组织成高并发流程。

### 1.24 Python 的 `list` 底层是链表吗

不是。

- CPython 的 `list` 底层是变长数组，不是链表
- 所以按索引访问很快
- 但头部插入、头部删除不高效

面试常见追问：

- 队列不要优先用 `list.pop(0)`，更适合用 `collections.deque`

### 1.25 MRO 是什么

- MRO 是方法解析顺序，决定多继承场景下属性和方法的查找顺序
- Python 使用 C3 线性化规则

面试里一般不会深挖，知道：

- 多继承查找顺序不是简单深度优先
- 可以通过 `类名.mro()` 查看

## 2. AI 应用开发岗位更容易追问的点

### 2.1 调大模型 API 属于什么类型任务

通常属于 I/O 密集型。

原因：

- 大部分时间在等待网络返回
- 所以并发调用模型接口时，`asyncio` 或多线程通常比多进程更合适

### 2.2 本地推理 / 大量文本清洗 / embedding 批处理怎么选并发模型

- 如果主要在等待网络、数据库、对象存储：优先协程或线程
- 如果主要在吃 CPU，比如大批量文本切分、复杂预处理、本地模型推理前后处理：优先多进程

### 2.3 为什么生成器在 AI 应用里常见

因为它适合“流式输出”。

例如：

- 大模型 token streaming
- 大文件逐行读取
- 批量数据分块处理

面试可答：

生成器可以边产出边消费，减少内存占用，也更适合流式接口。

### 2.4 为什么默认参数、浅拷贝在 AI 应用里容易埋坑

典型场景：

- `messages=[]` 导致不同请求共享上下文
- 浅拷贝对话状态后，内部嵌套字典还在共享，导致请求之间串数据

面试回答：

AI 应用经常处理会话状态、工具上下文、Prompt 模板，默认可变参数和浅拷贝非常容易导致脏状态。

### 2.5 为什么要熟悉上下文管理器

因为 AI 应用经常接触资源对象：

- 文件
- 锁
- 数据库连接
- 网络客户端

如果资源没及时释放，线上服务很容易出问题，比如：

- 句柄泄漏
- 连接池耗尽
- 文件未关闭

### 2.6 为什么很多 AI 服务更偏向异步框架

因为典型链路经常包含大量 I/O：

- 用户请求
- 模型接口调用
- 向量数据库检索
- 缓存读写
- 工具调用

这些步骤等待时间远高于 CPU 计算，所以异步模型能提升吞吐。

## 3. 面试快问快答

### Q1：`None` 判断为什么常写 `is None`，不写 `== None`

因为 `None` 是单例对象，判断身份更准确，风格上也更推荐 `is None`。

### Q2：为什么 `list.sort()` 返回 `None`

因为它是原地排序，设计上就是提醒你“这个操作修改了原列表”。

### Q3：字典为什么查找快

因为底层是哈希表，平均情况下查找、插入、删除效率都比较高。

### Q4：元组能不能作为字典 key

能，但前提是元组里的元素都可哈希。

### Q5：线程安全问题通常出现在什么地方

多个线程共享可变对象并同时修改时，容易出现竞态条件。

### Q6：协程是不是多线程

不是。协程通常是在单线程里完成高并发 I/O 调度。

### Q7：多线程为什么不适合 CPU 密集型

因为 CPython 里有 GIL，同一时刻只有一个线程执行 Python 字节码。

### Q8：什么时候用深拷贝

当对象有嵌套结构，而且你明确不希望内部子对象继续共享时。

## 4. 一段适合面试现场的总回答

如果让我总结 Python 在 AI 应用开发岗最重要的基础点，我会重点关注四类：

- 第一类是对象模型，比如可变与不可变、赋值与拷贝、`is` 和 `==`，因为这些会直接影响会话状态和数据处理是否串数据。
- 第二类是函数与作用域，比如默认参数、闭包、装饰器、`*args`/`**kwargs`，这些在写框架代码和工具封装时很常见。
- 第三类是迭代与资源管理，比如生成器、`with`、异常处理，这些和流式输出、文件处理、资源释放强相关。
- 第四类是并发模型，比如 GIL、线程、进程、协程，AI 应用里大量场景都是 I/O 密集型，请求编排通常更适合异步或线程，而 CPU 密集型处理更适合多进程。
