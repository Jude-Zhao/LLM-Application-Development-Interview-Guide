# 大模型八股文（AI 应用开发面试向）

> 整理时间：2026-03-17  
> 适用岗位：AI 应用开发、AI 产品研发、LLM 应用工程、智能体应用开发  
> 说明：这份内容按“应用开发岗”的面试深度整理，重点不是推公式，而是把概念、工程取舍、项目表达讲清楚。

---

## 1. 这个岗位应该掌握到什么深度

### 1.1 必会深度

你至少要能把下面这些问题讲清楚：

- 大模型为什么强，和传统 NLP / 传统机器学习有什么区别
- Transformer 是什么，为什么主流大模型都基于它
- 预训练、SFT、RLHF 分别在做什么
- Prompt、Few-shot、Chain-of-Thought 是什么
- Embedding、向量检索、RAG 分别解决什么问题
- Tool Calling / Function Calling 是什么，为什么 AI 应用离不开它
- 微调、LoRA、QLoRA 分别适合什么场景
- 幻觉为什么会出现，怎么缓解
- 如何做评测，如何看准确率、召回、延迟、成本

### 1.2 加分深度

如果能再讲到这些，面试表现会更稳：

- context window、token、KV cache 的基本含义
- top_k / top_p / temperature 对生成结果的影响
- RAG 的 chunk 切分、召回、rerank、生成链路
- 为什么很多 AI 服务更适合异步并发
- 结构化输出、JSON Schema、约束生成
- Prompt 缓存、流式输出、批处理、模型分级路由

### 1.3 不用硬啃太深的内容

对 AI 应用开发岗，一般不用把这些讲到论文推导级别：

- Attention 的矩阵推导
- RLHF 的 PPO 细节
- 各种位置编码公式
- 各种量化算法的数学细节
- 分布式训练和并行策略底层实现

一句话：

应用岗更看重你是否知道“为什么这么设计、线上怎么落地、遇到问题怎么排查”。

## 2. 面试高频背诵版

### 2.1 什么是大模型

大模型通常指参数规模很大、在海量数据上预训练、具备较强通用能力的基础模型。

它的特点是：

- 参数多
- 训练数据量大
- 能通过 Prompt 适配很多任务
- 不是只做单一分类或单一抽取任务

一句话回答：

大模型本质上是经过大规模预训练的通用基础模型，能通过上下文和提示词完成多任务。

### 2.2 大模型和传统 NLP 模型的区别是什么

传统 NLP 更像：

- 每个任务单独做一个模型
- 依赖人工特征或专门标注
- 泛化能力有限

大模型更像：

- 先预训练一个通用模型
- 再通过 Prompt、检索、工具、微调适配业务
- 可以一套模型处理问答、总结、抽取、分类、改写、代码等多种任务

一句话回答：

传统 NLP 是“一个任务一个模型”，大模型是“先有通用底座，再做任务适配”。

### 2.3 Transformer 是什么，为什么它成为主流

Transformer 是当前大模型最核心的架构之一。

你面试里不用背公式，知道这几点就够：

- 它以 self-attention 为核心
- 更容易并行训练
- 更擅长建模长距离依赖
- 大量主流大模型都建立在 Transformer 或其变体之上

根据《Attention Is All You Need》，Transformer 使用堆叠的 self-attention 和前馈网络来完成序列建模。

一句话回答：

Transformer 之所以成为主流，是因为它比早期 RNN 更容易并行、更适合大规模训练，也更适合处理长上下文依赖。

### 2.4 Attention 是什么

可以简单理解为：

- 模型在生成某个 token 时，会动态关注输入里更相关的位置

Self-attention 可以理解为：

- 序列内部各个位置彼此“看一眼”，决定当前该关注哪些信息

一句话回答：

Attention 的核心作用是让模型在处理当前内容时，动态聚焦更相关的上下文。

### 2.5 Token 是什么

Token 不是“字”也不是“词”，而是模型处理文本时使用的基本切分单元。

注意：

- 英文单词可能被拆成多个 token
- 中文通常一个字或词会映射成若干 token，具体取决于 tokenizer
- 模型的输入和输出成本，很多都按 token 计费或计长

一句话回答：

Token 是模型实际处理文本的最小单位，和肉眼看到的字词不完全一一对应。

### 2.6 Context Window 是什么

- Context window 就是模型一次能看到的总上下文长度上限
- 包括输入 token 和输出 token

对应用开发很重要，因为：

- 放不下就会截断
- 长上下文会带来更高延迟和更高成本
- 不等于“放进去越多越好”

一句话回答：

Context window 是模型一次能处理的上下文容量上限，越长不一定越好，因为会影响成本、延迟和效果。

### 2.7 预训练、SFT、RLHF 分别是什么

#### 预训练

- 在海量文本上做通用训练
- 让模型学会语言规律、知识分布、通用表达和任务迁移能力

#### SFT

- Supervised Fine-Tuning，监督微调
- 用“输入-理想输出”样本教模型更稳定地按目标风格回答

#### RLHF

- Reinforcement Learning from Human Feedback
- 用人类偏好数据继续对齐，让模型更符合“有帮助、无害、遵循指令”的目标

根据 InstructGPT 论文，典型流程是先做 SFT，再基于人类排序偏好继续优化。

一句话回答：

预训练让模型“会说话”，SFT 让模型“更会按要求说”，RLHF 让模型“更像人想要的那样说”。

### 2.8 什么是 In-Context Learning

In-context learning 指的是：

- 不改模型参数
- 只通过上下文里的说明、示例、格式约束，让模型在当前请求中学会任务

GPT-3 论文里常见三种说法：

- zero-shot：只给任务说明，不给示例
- one-shot：给一个示例
- few-shot：给少量示例

一句话回答：

In-context learning 就是不训练模型，只靠当前上下文中的说明和示例完成任务适配。

### 2.9 Prompt Engineering 是什么

Prompt Engineering 就是设计输入，让模型更稳定地产出目标结果。

常见手段：

- 角色设定
- 明确任务边界
- 规定输出格式
- 给 few-shot 示例
- 提供反例或约束

一句话回答：

Prompt Engineering 的核心不是“玄学咒语”，而是通过清晰任务描述和示例降低模型歧义。

### 2.10 Chain-of-Thought 是什么

Chain-of-Thought，简称 CoT，可以理解为：

- 引导模型按中间推理步骤来回答复杂问题

NeurIPS 2022 的 CoT 论文表明，给出中间推理示例，能明显提升复杂推理任务表现。

但应用开发岗面试要注意两点：

- CoT 更适合复杂推理，不是所有任务都需要
- 让模型输出大段思维链，可能增加延迟、成本，甚至带来不稳定性

一句话回答：

CoT 是通过显式中间步骤提升复杂任务推理效果，但不是所有场景都应该默认开启。

### 2.11 Temperature 和 Top-p 是什么

#### Temperature

- 控制采样随机性
- 越低越稳定，越高越发散

#### Top-p

- 只在累计概率达到阈值的候选 token 里采样

面试常见回答：

- 做分类、抽取、结构化生成时，温度一般低一点
- 做创作、文案、发散 brainstorming 时，可以适当高一点

一句话回答：

Temperature 更像控制“发散程度”，Top-p 更像控制“采样候选范围”。

### 2.12 什么是幻觉

幻觉就是：

- 模型生成了看起来很合理，但事实错误、来源不存在、或不受输入支持的内容

常见原因：

- 训练知识不是实时的
- 生成目标本身是“预测下一个 token”，不是“保证事实为真”
- 检索没召回到正确信息
- Prompt 不清晰

一句话回答：

幻觉本质上是语言生成能力强于事实校验能力的表现。

### 2.13 如何缓解幻觉

常见方法：

- 用 RAG 接外部知识
- 要求基于给定材料作答
- 做引用或证据约束
- 用结构化输出减少随意发挥
- 对关键结果加规则校验或二次校验
- 对高风险场景做人审或工具校验

一句话回答：

缓解幻觉的核心不是“更会写 Prompt”，而是把检索、约束、校验一起加进系统。

### 2.14 Embedding 是什么

Embedding 是把文本变成向量表示。

它的核心用途不是“生成答案”，而是：

- 做相似度搜索
- 做召回
- 做聚类
- 做推荐
- 做分类

官方文档通常会把它描述为“衡量文本相关性”的向量表示。

一句话回答：

Embedding 的作用是把语义映射到向量空间，让机器能做相似度计算和检索。

### 2.15 RAG 是什么

RAG 是 Retrieval-Augmented Generation，检索增强生成。

它的核心思想是：

- 先从外部知识库检索相关内容
- 再把检索结果连同用户问题一起交给大模型生成答案

RAG 论文的关键思想可以概括为：

- 把参数化记忆和非参数化记忆结合起来

一句话回答：

RAG 不是让模型“自己想”，而是让模型“先查再答”。

### 2.16 为什么 RAG 对 AI 应用很重要

因为业务知识经常是：

- 私有的
- 高频更新的
- 不在模型训练集里的

比如：

- 企业文档
- 产品手册
- SOP
- 代码库
- 政策和知识库

一句话回答：

RAG 让模型具备访问外部最新知识的能力，是很多企业 AI 应用落地的基础设施。

### 2.17 RAG 的基本流程是什么

最常见链路：

1. 文档清洗
2. 文本切块
3. 生成 embedding
4. 向量检索召回
5. 可选 rerank
6. 把召回内容拼进 prompt
7. 生成答案
8. 可选引用、校验、评测

面试里不要只说“向量库 + 大模型”，而要能讲清楚整个链路。

### 2.18 RAG 和微调有什么区别

#### RAG

- 主要解决知识接入问题
- 适合资料经常更新
- 不改模型参数

#### 微调

- 主要解决行为风格或特定任务模式问题
- 适合固定格式、固定风格、固定任务
- 需要训练和评测成本

一句话回答：

RAG 更偏“补知识”，微调更偏“改行为”。

### 2.19 什么场景下优先选 RAG

- 你的知识经常更新
- 你的知识是企业私域数据
- 你需要可追溯引用
- 你不想频繁重新训练模型

### 2.20 什么场景下考虑微调

- 输出格式长期稳定
- 任务模式固定
- Prompt 很长、样例很多，推理成本高
- 通用模型很难稳定学会某种领域风格
- 需要提升某一类任务的一致性

### 2.21 LoRA 是什么

LoRA 是一种参数高效微调方法。

核心思想：

- 冻结原模型大部分参数
- 只训练少量新增的低秩参数

LoRA 论文指出，它通过在 Transformer 层注入可训练的低秩矩阵，大幅减少可训练参数和显存需求。

一句话回答：

LoRA 的本质是“少改参数、低成本适配”。

### 2.22 QLoRA 是什么

QLoRA 可以理解为：

- 在量化模型基础上继续做 LoRA 微调
- 进一步降低显存占用

QLoRA 论文给出的典型卖点是：

- 能把很大的模型在单卡较小显存条件下完成微调

一句话回答：

QLoRA 是 LoRA 的低显存版本，适合资源有限时做大模型微调实验。

### 2.23 量化是什么

量化就是：

- 用更低精度表示模型参数，比如 8bit、4bit

好处：

- 降低显存
- 降低部署成本
- 提高推理吞吐

代价：

- 可能损失部分精度
- 不同量化方案工程复杂度不同

一句话回答：

量化本质是在效果和资源之间做折中。

### 2.24 Tool Calling / Function Calling 是什么

Tool Calling 是让模型不只会“说”，还会“调用外部能力”。

比如让模型调用：

- 搜索
- 数据库
- 天气接口
- 订单系统
- 内部工作流

官方文档里，典型流程是：

1. 给模型可用工具定义
2. 模型决定是否发起工具调用
3. 应用侧执行工具
4. 把工具结果回传给模型
5. 模型再生成最终回复

一句话回答：

Tool Calling 的价值是把大模型从“会聊天”升级为“能操作系统和数据”的执行中枢。

### 2.25 Structured Output 是什么

Structured Output 指的是：

- 让模型按预设结构输出，比如固定 JSON Schema

价值：

- 更适合系统对接
- 更容易程序解析
- 减少字段缺失和格式漂移

一句话回答：

Structured Output 的核心是让模型输出从“给人看”变成“给程序安全消费”。

### 2.26 Tool Calling 和 Structured Output 的区别

- Tool Calling：重点是让模型决定“要不要调用工具”和“怎么调”
- Structured Output：重点是让模型按约定结构返回结果

一句话回答：

前者偏执行，后者偏结果格式约束。

### 2.27 Agent 是什么

面试里推荐你这样回答：

- Agent 不是单独一种模型
- 它更像“模型 + 工具 + 记忆 + 规划 + 执行流程”的组合系统

也就是说：

- 模型负责理解和决策
- 工具负责查数据和执行动作
- 工作流负责多步任务推进

一句话回答：

Agent 本质上是把大模型接到工具链和执行链上的系统能力，不只是一次性文本生成。

### 2.28 为什么 AI 应用经常要做流式输出

因为用户对“首字时间”很敏感。

流式输出的价值：

- 用户更快看到结果
- 交互体验更好
- 适合长回答和长推理过程

一句话回答：

流式输出不一定减少总耗时，但通常能明显改善用户体感延迟。

### 2.29 为什么很多 AI 应用偏异步架构

因为一条请求链路里经常有大量 I/O：

- 模型 API
- 检索
- 数据库
- 缓存
- 工具调用

这类场景等待时间多于 CPU 计算，所以异步或并发编排更常见。

### 2.30 大模型应用怎么做评测

至少要分两类：

#### 离线评测

- 准确率
- 召回率
- F1
- BLEU / ROUGE（有些场景会用）
- 主观打分
- LLM-as-a-judge（要谨慎）

#### 在线评测

- 点击率
- 完成率
- 用户满意度
- 人工复核通过率
- 业务转化
- 延迟
- 成本

一句话回答：

大模型评测不能只看“像不像”，还要看业务指标、稳定性、成本和风险。

### 2.31 大模型应用怎么优化延迟和成本

常见思路：

- 缩短 prompt
- 做缓存
- 使用更小模型处理简单任务
- 复杂任务再升级到大模型
- 流式输出
- 并行检索或并行工具调用
- 限制返回长度
- 做 prompt 模板复用

一句话回答：

延迟和成本优化的本质，是让“贵的模型只做真正需要它做的事”。

## 3. AI 应用开发岗最容易被追问的知识链

### 3.1 从用户问题到模型回答，中间一般经历什么

可以答成一条工程链路：

1. 接收用户输入
2. 做输入清洗和安全检查
3. 判断是否需要检索、调用工具或走特定工作流
4. 构造 prompt 或结构化请求
5. 调模型
6. 对结果做解析、校验、后处理
7. 返回给前端，并记录日志、埋点评测

如果面试官继续追问，你可以补：

- 是否要做会话记忆压缩
- 是否要做多轮上下文裁剪
- 是否要做缓存
- 是否要做失败重试

### 3.2 为什么 RAG 项目效果差，不一定是模型不行

RAG 项目常见瓶颈：

- 文档切块不合理
- embedding 模型不匹配
- 召回不到位
- top_k 不合理
- rerank 缺失
- 拼接上下文过长，把噪声也塞给模型了
- Prompt 没强约束“只能基于资料回答”

一句话：

RAG 效果差，很多时候问题不在生成，而在检索链路。

### 3.3 为什么只会写 Prompt 不够

因为线上问题往往不是单纯的提示词问题，而是：

- 数据问题
- 检索问题
- 工具定义问题
- 输出约束问题
- 评测口径问题
- 缓存和并发问题

面试里这样答会更像工程岗，而不是只会调模型的人。

### 3.4 为什么结构化输出很重要

因为真实系统不是只给人看答案，还要：

- 让后端解析
- 触发工作流
- 存数据库
- 调下游接口

如果输出结构不稳定，就很难接工程系统。

### 3.5 为什么工具定义要尽量清晰

因为工具定义本质上也是 prompt 的一部分。

工具描述不清，常见后果：

- 不该调用时乱调用
- 该调用时不调用
- 参数填错
- 多个工具混淆

官方文档里也强调：

- 工具数量不要一上来给太多
- 工具定义要直观，减少模型理解负担

## 4. 面试快问快答

### Q1：为什么说大模型是“概率生成”，不是“数据库查询”

因为它本质是在当前上下文下预测下一个 token，不是直接去事实库里精确检索答案。

### Q2：为什么模型会一本正经胡说八道

因为它优化目标是生成高概率、自然流畅的文本，而不是天然保证事实正确。

### Q3：Few-shot 一定比 zero-shot 好吗

不一定。任务简单时 zero-shot 就够；示例太差、太长或太噪，也可能让效果更差。

### Q4：RAG 能完全消除幻觉吗

不能。它只能显著缓解，最终还取决于召回质量、上下文约束和后处理校验。

### Q5：Embedding 和大模型生成是同一回事吗

不是。Embedding 主要用于表示和检索，生成模型主要用于输出文本。

### Q6：RAG 和搜索引擎一样吗

不完全一样。RAG 通常是“先检索，再把结果交给生成模型组织成答案”。

### Q7：微调是不是一定比 Prompt 好

不是。微调成本更高，适合稳定场景；很多知识更新类问题优先还是 RAG。

### Q8：LoRA 为什么常见

因为它训练成本和显存需求更低，更适合业务团队做小步快跑的微调实验。

### Q9：Agent 一定比单轮问答强吗

不一定。任务简单时，单轮问答更稳、更便宜。Agent 适合多步任务和外部系统协作。

### Q10：温度越低越好吗

不一定。抽取、分类、结构化场景通常更适合低温；创作类任务可以适当提高。

### Q11：大上下文是不是一定更强

不一定。上下文太长会增加噪声、延迟和成本，也可能降低有效信息密度。

### Q12：AI 应用开发最重要的能力是什么

不是只会调 API，而是能把模型、数据、检索、工具、评测和业务链路一起打通。

## 5. 一段适合面试现场的总回答

如果让我总结 AI 应用开发岗最需要掌握的大模型知识，我会分成五层：

- 第一层是模型基础，要理解 Transformer、token、上下文窗口、预训练和对齐流程，这样知道模型能力边界在哪里。
- 第二层是任务适配，要理解 Prompt、few-shot、结构化输出、工具调用，因为很多业务场景不是纯聊天，而是要接系统。
- 第三层是知识增强，要理解 embedding、向量检索和 RAG，因为企业应用大量依赖私有和实时知识。
- 第四层是模型定制，要知道什么时候该用 Prompt，什么时候该用 RAG，什么时候才值得做 LoRA 或 SFT。
- 第五层是工程落地，要能讲清评测、延迟、成本、缓存、并发和稳定性，这部分决定系统能不能上线。

## 7. AI 应用开发岗的“合适深度”建议

### 7.1 面试回答应该达到的层次

推荐你用这个标准准备：

- 第一层：能解释概念
- 第二层：能解释为什么
- 第三层：能解释在项目里怎么选型
- 第四层：能解释常见失败点和排查思路

例如被问到 RAG，不要只答：

“RAG 就是先检索再生成。”

更好的回答应该是：

“RAG 用于把外部知识接入模型，核心链路通常是切块、embedding、召回、可选 rerank，再把证据拼进 prompt 生成答案。它更适合知识经常更新、需要引用和可追溯的业务场景。效果不好的时候，我会优先检查 chunk 策略、召回质量、top_k 和 prompt 约束，而不只是怀疑模型本身。”

### 7.2 你不需要卷到的深度

对应用岗来说，下面这些知道名字和大意就够了：

- Flash Attention
- MoE
- speculative decoding
- rope / alibi 等位置编码细节
- 张量并行、流水并行

知道它们存在是加分，但一般不会是主战场。
