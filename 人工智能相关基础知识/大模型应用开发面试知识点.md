# 大模型应用开发面试知识点

## 一、基础概念

### 1.1 大语言模型（LLM）
- **定义**：基于 Transformer 架构的大规模预训练语言模型
- **代表模型**：GPT 系列、Claude、LLaMA、通义千问、文心一言等
- **核心能力**：文本生成、推理、代码编写、对话理解

### 1.2 Token 与 Tokenizer
- **Token**：模型处理的最小单位，可能是单词、子词或字符
- **Tokenizer**：将文本转换为 token 的工具
- **估算**：1 Token ≈ 4 个英文字符 ≈ 1-2 个中文字符
- **上下文窗口**：模型能处理的最大 token 数量

### 1.3 Prompt Engineering
- **Few-shot Prompting**：通过示例引导模型输出
- **Chain-of-Thought (CoT)**：引导模型展示推理过程
- **Zero-shot**：不给示例，直接描述任务
- **System Prompt**：设定模型角色和行为规范

## 二、RAG（检索增强生成）

### 2.1 核心原理
```
用户问题 → 检索 → 增强上下文 → LLM生成 → 最终回答
```

### 2.2 向量数据库
| 数据库 | 特点 |
|--------|------|
| Milvus | 开源、高性能、支持混合检索 |
| Pinecone | 云原生、易用性好 |
| Chroma | 轻量级、适合本地开发 |
| Weaviate | 原生支持混合搜索 |

### 2.3 Embedding 模型
- **常用模型**：text-embedding-ada-002、text-embedding-3-small、BGE、M3E
- **评估指标**：召回率、相关性、准确性

### 2.4 分块策略（Chunking）
- **固定大小分块**：简单直接，可能截断语义
- **句子级别分块**：保持句子完整性
- **段落级别分块**：保留上下文
- **语义分块**：根据内容语义动态分块

### 2.5 常见优化
- **混合检索**：向量检索 + 关键词检索（BM25）
- **重排序（Rerank）**：对检索结果二次排序
- **Query 改写**：优化用户问题以提高检索效果
- **Self-RAG**：让模型自我评估检索结果的相关性

## 三、Agent（智能体）

### 3.1 Agent 核心组件
```
Agent = LLM + 工具（Tools）+ 规划（Planning）+ 记忆（Memory）
```

### 3.2 工具调用（Function Calling）
- **定义**：让 LLM 能够调用外部工具或 API
- **常见框架**：OpenAI Function Calling、LangChain Tools、ReAct
- **工具设计原则**：
  - 工具名称清晰明确
  - 参数描述完整
  - 返回结果结构化

### 3.3 ReAct 模式
```
Thought → Action → Observation → Thought → ...
思考 → 行动 → 观察 → 思考 → ...
```
- **Thought**：分析当前情况，决定下一步
- **Action**：执行具体动作（调用工具）
- **Observation**：获取行动结果

### 3.4 规划能力
- **任务分解**：将复杂任务拆分为子任务
- **自我反思**：评估执行结果，必要时重新规划
- **常见框架**：CoT、ToT（Tree of Thoughts）、GoT（Graph of Thoughts）

### 3.5 记忆模块
| 类型 | 特点 | 应用场景 |
|------|------|----------|
| 短期记忆 | 会话级，仅当前对话有效 | 对话上下文 |
| 长期记忆 | 持久化，可跨会话 | 用户偏好、历史交互 |
| 工作记忆 | 当前任务相关信息 | 问题解决过程 |

## 四、大模型微调

### 4.1 全量微调（Full Fine-tuning）
- **优点**：效果好，能充分适应新任务
- **缺点**：成本高，容易过拟合，灾难性遗忘

### 4.2 参数高效微调（PEFT）
| 方法 | 原理 | 优点 |
|------|------|------|
| LoRA | 注入低秩矩阵 | 参数量小，训练快 |
| QLoRA | LoRA + 量化 | 更低显存 |
| Adapter | 添加适配器层 | 可训练参数少 |
| Prefix Tuning | 添加可学习前缀 | 不改变原模型 |

### 4.3 LoRA 核心公式
```
W = W₀ + ΔW = W₀ + BA
其中 B ∈ R^(d×r), A ∈ R^(r×k), r << min(d,k)
```

### 4.4 训练数据格式
- **SFT（有监督微调）**：指令-回答对
- **RLHF**：
  - SFT（监督微调）
  - Reward Model（奖励模型）
  - PPO（强化学习优化）

## 五、Prompt Engineering 进阶

### 5.1 结构化 Prompt
```markdown
# 角色
你是一个专业的Python后端开发工程师。

# 技能
- 熟练掌握Django、Flask框架
- 熟悉MySQL、Redis数据库
- 理解RESTful API设计原则

# 输出格式
1. 代码实现
2. 关键点解释
3. 可能的优化建议
```

### 5.2 常用技巧
- **角色设定**：明确 AI 扮演的角色
- **格式约束**：指定输出格式（JSON、Markdown 等）
- **示例引导**：提供 few-shot 示例
- **思维链**：引导模型分步骤思考
- **安全边界**：设置输出限制和过滤规则

### 5.3 Temperature 与 Top-p
- **Temperature**：控制随机性
  - 0.0-0.3：确定性输出，适合精确任务
  - 0.7-1.0：创意输出，适合生成任务
- **Top-p（核采样）**：控制 token 候选集大小
- **Top-k**：限制每次选择的 token 数量

## 六、模型部署与优化

### 6.1 推理优化技术
- **量化（Quantization）**：
  - INT8/INT4 量化减少显存占用
  - AWQ、GPTQ、GGUF 等量化方法
- **KV Cache**：缓存键值对，加速生成
- **Batching**：批量处理提高吞吐量
- **推测解码（Speculative Decoding）**：小模型推测，大模型验证

### 6.2 部署框架
| 框架 | 特点 |
|------|------|
| vLLM | 高吞吐量，PagedAttention |
| TensorRT-LLM | NVIDIA优化，高性能 |
| Ollama | 轻量级，本地部署 |
| FastChat | 开源，支持多种模型 |

### 6.3 API 调用
```python
# OpenAI 风格调用示例
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": "你是一个助手"},
        {"role": "user", "content": "你好"}
    ],
    temperature=0.7,
    max_tokens=1000
)
```

## 七、评估与质量保证

### 7.1 评估维度
| 维度 | 说明 | 评估方法 |
|------|------|----------|
| 准确性 | 回答是否正确 | 人工评估、Benchmark |
| 相关性 | 是否切题 | 用户反馈、指标计算 |
| 流畅性 | 表达是否通顺 | 语言质量指标 |
| 安全性 | 是否有害内容 | 安全检测 |

### 7.2 常见 Benchmark
- **MMLU**：多任务语言理解
- **HumanEval**：代码生成
- **BBH**：复杂推理
- **C-Eval**：中文能力评估

### 7.3 Red Teaming
- **定义**：故意寻找模型的弱点和漏洞
- **常见攻击**：
  - Prompt Injection（提示注入）
  - Jailbreak（越狱）
  - 角色扮演绕过限制
  - 恶意内容生成

### 8.2 LangChain 核心概念
```python
# 简单示例
from langchain_openai import ChatOpenAI
from langchain.prompts import PromptTemplate

llm = ChatOpenAI(model="gpt-4")
prompt = PromptTemplate.from_template("{topic}的三个关键点是什么？")
chain = prompt | llm
response = chain.invoke({"topic": "大模型"})
```
